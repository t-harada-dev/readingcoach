# ReadingCoach 現行アーキテクチャ整理

## 目的

- 現在の主導線を明確にする
- 新実装と旧実装の境界を可視化する
- 削除候補と、削除前に埋めるべき欠落を特定する

## 結論

- 現在の主導線は `FocusCore -> FocusBookPicker -> ActiveSession` の新実装
- データの正本として寄せたい先は `PersistenceBridge.ts` 経由の SwiftData
- `ExecutionScreen` 系と `context.tsx` 系は旧 AsyncStorage 導線として残存している
- 通知・Live Activity は JS 側の契約はあるが、現ワークスペース上では iOS ネイティブ実装の実体が確認できない

## 画面遷移

### 現在の主導線

```text
App.tsx
  -> AppInitProvider
    -> runReconcileThenNotifyReady("app_launch")
  -> NavigationContainer
    -> FocusCoreScreen (initialRoute)
      -> FocusBookPickerScreen
      -> ActiveSessionScreen
      -> ReserveScreen
      -> AddBookScreen
```

### 新UIの遷移詳細

```text
FocusCoreScreen
  - 起動/復帰時に reconcile を実行
  - 当日 plan を取得
  - plan.bookId から book を取得
  - 15分 or 1分の開始 CTA を表示
  - 「本を切り替える」で FocusBookPicker へ

FocusBookPickerScreen
  - books 一覧を表示
  - 選択時に upsertPlan(planId, planDate, bookId)
  - 1日1回制限カウントを加算
  - 戻る

ActiveSessionScreen
  - StartSessionUseCase の返却値を受け取る
  - endTimeISO までの残り時間を表示
  - 完了後は表示上 Completed に切り替わる
```

### 旧導線

```text
ExecutionScreen
  -> ReserveScreen
  -> AddBookScreen
```

- `ExecutionScreen` には `旧ホーム` コメントがあり、比較用・退避用の位置づけ
- こちらは `useApp()` と `storage.ts` を使う AsyncStorage 中心の別系統

## データフロー

### 新実装の主経路

```text
App launch
  -> AppInitProvider
  -> runReconcileThenNotifyReady()
  -> persistenceBridge.reconcilePlans()
  -> iOS PersistenceBridge.swift
  -> PersistenceStore.reconcilePlans()
  -> SwiftData (Book / ExecutionPlan / SessionLog / Settings)
  -> 結果を JS へ返す
  -> notificationBridge.ready()

FocusCoreScreen 表示
  -> runReconcilePlansUseCase("foreground")
  -> persistenceBridge.getPlanForDate()
  -> persistenceBridge.getBook()
  -> UI 描画

セッション開始
  -> runStartSessionUseCase()
  -> persistenceBridge.startSession()
  -> PersistenceStore.startSession()
  -> SessionLog 作成 + Plan を active 化
  -> liveActivityBridge.startSession()
  -> ActiveSessionScreen 遷移
```

### 旧実装の経路

```text
ExecutionScreen / ReserveScreen / AddBookScreen
  -> useApp()
  -> context.tsx
  -> storage.ts
  -> AsyncStorage
```

### 境界上の注意点

- 新実装は `BookDTO` / `DailyExecutionPlanDTO` / `SessionLogDTO` を使う
- 旧実装は `Book` / `Reservation` / `ExecutionLogEntry` を使う
- 同じ「本」「実行」の概念でも型・保存先・選定ロジックが別物

## 削除候補

### 優先度: 高

- `app/src/screens/ExecutionScreen.tsx`
  - 既に `旧ホーム` 扱い
  - 新主導線とデータソースが異なる

- `app/src/context.tsx`
  - `ExecutionScreen` / `ReserveScreen` / `AddBookScreen` のためだけに残っている
  - 現行主導線はこの context を実質使っていない

- `app/src/storage.ts`
  - 旧導線専用
  - SwiftData 正本化の方針と二重管理になる

- `app/src/types.ts`
  - 旧導線専用型
  - DTO 群と二重定義になっている

- `app/src/selectTodayBook.ts`
  - 旧導線の「今日の1冊」決定ロジック
  - 現在は native reconcile / native plan が正本

### 優先度: 中

- `app/src/notifications.ts`
  - Expo Notifications による旧予約通知
  - `NotificationBridge` に寄せるなら役割が重複する

- `app/src/screens/ReserveScreen.tsx`
  - 旧 reservation モデル前提
  - ただし新実装で「明日の予約」機能をまだ完全代替できていないなら即削除は危険

- `app/src/screens/AddBookScreen.tsx`
  - 旧 context 経由で book を追加している
  - 新規追加 UI 自体は必要なので、削除ではなく `persistenceBridge.saveBook()` ベースへ移植してから差し替えるべき

## 削除前に埋めるべき欠落

### 1. AddBook の新実装化

- いまの `AddBookScreen` は UI として必要
- ただし保存先を `context.addBook()` から `persistenceBridge.saveBook()` に置き換える必要がある

### 2. Reserve の新実装化

- いまの `ReserveScreen` は `Reservation` を AsyncStorage に保存している
- 新実装では `DailyExecutionPlan` / 通知再整合に寄せる必要がある

### 3. Notification / Live Activity の native 実体確認

- JS 側 bridge はある
- ただし現ワークスペースでは対応する iOS 実装本体が見えない
- 削除整理前に「存在するが未共有」なのか「未実装」なのかを確定させる必要がある

## 推奨整理順

1. `AddBookScreen` を `persistenceBridge.saveBook()` ベースへ移行
2. `ReserveScreen` を `ExecutionPlan` ベースへ移行
3. `App.tsx` から `ExecutionScreen` を外す
4. `context.tsx` / `storage.ts` / `types.ts` / `selectTodayBook.ts` を削除
5. `notifications.ts` を `NotificationBridge` へ統合するか削除する
