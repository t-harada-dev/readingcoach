# ReadingCoach 移行計画書（2026-03-10）

## 目的

* 新旧二重化を解消し、正本を一本化する
* MVP を削らずに、状態機械と OS 接点を同一中核モデルへ収束させる
* 旧導線の削除条件と順序を明確化する

## 中核モデル（固定）

* 正本: `SwiftData`
* 状態主体: `DailyExecutionPlan`
* 行動記録: `SessionLog`
* 読書対象: `Book`
* OS 接点 read model: `WidgetSnapshotStore` 相当（shared container）
* 起点統一: `ReconcilePlansUseCase` と開始系 use case

## 現在の主導線

```text
App.tsx
  -> AppInitProvider
    -> runReconcileThenNotifyReady("app_launch")
  -> FocusCoreScreen (initialRoute)
    -> FocusBookPickerScreen
    -> ActiveSessionScreen
    -> ReserveScreen
    -> AddBookScreen
```

## 問題の本質

* 新導線は `PersistenceBridge` + SwiftData 前提
* 旧導線（`ExecutionScreen / ReserveScreen / AddBookScreen` + `context.tsx`）は AsyncStorage 書き込みを継続
* 同概念が別型・別保存先で二重化している
* 画面が business rule を一部保持しており、use case 境界が不統一

## 移行ポリシー

1. 旧系への新規 write を停止する
2. screen から直接保存せず use case 経由へ統一する
3. one-shot migration（AsyncStorage -> SwiftData）が必要な場合のみ実行する
4. 二重書き込み期間を作らない
5. cutover criteria 達成後に旧ファイルを削除する

## レイヤ責務（移行後）

### Screen

* 入力受付と表示のみ
* business rule を持たない

### Use Case

* `AddBookUseCase`
* `ReservePlanUseCase`
* `ReconcilePlansUseCase`
* `StartSessionUseCase`
* `CompleteSessionUseCase`
* `NotificationResponseHandler`
* `WidgetStartSessionUseCase`
* `AppIntentStartSessionUseCase`
* `RehabRecoveryUseCase`

### Bridge / Native

* `PersistenceBridge` が Book / Plan / Session / Settings / Entitlement の唯一の write 境界
* `NotificationBridge` / `LiveActivityBridge` / Widget extension は shared read model を参照

## フェーズ別移行計画

### Phase 1: 正本一本化

* `AddBookScreen` を `AddBookUseCase` + `persistenceBridge.saveBook()` へ移行
* `ReserveScreen` を `ReservePlanUseCase` + `upsertPlan` へ移行（`Reservation` 廃止）
* `ExecutionScreen` を stack から除去
* 旧 `context.tsx` 系への write 呼び出しを全停止

### Phase 2: OS 一等地の実体固定

* 通知（定刻 / 再通知 / 応答）
* Widget（表示 / 開始）
* Live Activity（開始 / 更新 / 終了）
* App Intents（開始）
* 通知応答・Widget・App Intents の全経路で `Reconcile` 先行を保証

### Phase 3: 状態機械完成

* `continuous_missed_days` による CTA 切替
* Strategic Bypass
* `normal_15m / rescue_5m / rescue_3m / book_fetch_1m` の統一
* `prep_success` の継続維持と進捗分離

### Phase 4: 事業要素接続

* `EventLog` 必須化
* 資産化表示
* entitlement（one-time 優先、将来 subscription）
* Cloud Sync（Local First を維持）

## Cutover Criteria（削除前提条件）

以下を満たした時点で旧導線を削除可能とする。

* AddBook / Reserve が use case + SwiftData 経由で動作する
* `ExecutionScreen` 未使用で主要導線に参照がない
* 通知 / Widget / App Intents / app 内開始の全経路で状態遷移が一致する
* `ReconcilePlansUseCase` が全起点で先行する
* EventLog が必須イベントを記録する

## 旧ファイル削除条件

### 即時削除候補

* `app/src/screens/ExecutionScreen.tsx`（stack 除去完了後）

### 移植後削除候補

* `app/src/context.tsx`
* `app/src/storage.ts`
* `app/src/types.ts`
* `app/src/selectTodayBook.ts`
* `app/src/notifications.ts`
* 旧 `ReserveScreen` / 旧 `AddBookScreen` 実装

## OS Surface Definition of Done

### Notification

* 定刻通知が出る
* 通知から開始できる
* 開始時に関連通知が整理される

### Widget

* shared snapshot から今日状態を表示できる
* Widget から開始できる

### Live Activity

* セッション開始で表示
* 進行中に更新
* 完了/中断で終了

### App Intents

* 開始アクションが動作
* 実行前に `Reconcile` が先行

## 検証ケース（最低）

* app 起動開始
* 通知応答開始
* Widget 開始
* App Intent 開始
* 復帰時 reconcile
* 連続未達 3 日 / 7 日
* `prep_success` の継続維持と進捗非加算

## 現時点の判断

* 旧導線の write 停止を最優先に進める
* 旧ファイル削除は cutover criteria 達成後に実施する
* MVP を狭めずに前進するには、機能追加より先に正本・状態機械・OS 接点の一本化を完了させる
