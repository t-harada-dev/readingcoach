# 積読コーチ 異常系・縮退運転定義書

## 0. 目的

本書は、積読コーチにおける異常系・欠損・外部依存失敗・端末制約・状態不整合を整理し、**どの異常でも「今日の読書達成」を最大限止めない**ための縮退運転方針を定義する。  
対象は MVP の実装・UI・QA・受入試験である。

本書の役割は次の 4 点。

1. 想定すべき異常系を網羅的に列挙する
2. 異常時の優先順位を固定する
3. UI 表示・フォールバック・記録・復旧方針を統一する
4. 実装判断で「止める / 止めない」の基準を明確にする

---

## 1. 最上位原則

### 1.1 コア原則

- 最優先は**その日の読書達成を成立させること**
- 外部 API・書影・価格情報・同期・課金確認は、読書執行より優先しない
- 異常は可能な限り**局所化**し、執行フロー全体へ波及させない
- ユーザーに見せる文言は非詰問・非罪悪感・実務的に保つ
- 「壊れたので何もできません」は最後の最後まで避ける

### 1.2 異常時の優先順位

異常時は次の順で守る。

1. 今日のセッション開始・継続・完了
2. 当日計画の整合
3. 通知・Widget・App Intents の一貫性
4. 本追加・本変更・書誌検索
5. 書影・価格・ページ数など補助情報
6. 資産化表示
7. 同期・課金状態の最新性
8. 分析ログ

### 1.3 基本ルール

- **検索系が壊れても既存蔵書の読書は止めない**
- **書影がなくても本選択は止めない**
- **価格やページ数がなくても資産化以外は止めない**
- **進捗バー未設定・OFF・入力未実施でも主導線と達成体験を止めない**
- **同期や課金確認が遅れても local の基本利用は止めない**
- **通知が失敗してもアプリ内主導線は残す**
- **OS surface が一部死んでも app 内開始は生かす**

## 2. 異常レベル定義

### Level A: コア執行阻害

例:
- 当日 plan が取れない
- セッション開始できない
- active session が復旧不能
- DailyExecutionPlan が破損している

扱い:
- 最優先で復旧または再生成を試みる
- 失敗時は「今日のセッションを作り直す」縮退導線を出す
- 必ず EventLog に残す

### Level B: OS 接点阻害

例:
- 通知が予約されない
- Widget が stale
- App Intent から開始できない
- Live Activity が更新されない

扱い:
- app 内の主導線は維持する
- 表示・起点だけを縮退させる
- 次回 reconcile で回復を試みる

### Level C: 登録・検索阻害

例:
- 書籍検索 timeout
- 書影欠損
- metadata 欠損
- API 429 / 5xx

扱い:
- 手入力 fallback
- placeholder 表示
- 欠損許容で登録継続

### Level D: 補助価値阻害

例:
- 資産化額が出ない
- CloudKit 遅延
- entitlement 最新化遅延
- EventLog 書き込み失敗

扱い:
- 本体体験は継続
- 補助表示だけ抑制・遅延

## 3. 異常系一覧（網羅版）

## 3.1 初回設定・本追加

### E-01 書籍検索が timeout する

発生例:
- Google Books API 応答遅延
- モバイル回線不安定

システム対応:
- 既定 timeout 後に 1 回だけ自動 retry
- それでも失敗なら手入力導線へ切替
- spinner を長時間出し続けない

UI 表示:
- 「通信エラーにより検索できませんでした。手入力で登録しますか？」

止めてはいけないもの:
- 本登録そのもの

記録:
- `book_search_timeout`
- `book_search_manual_fallback`

### E-02 検索結果 0 件

システム対応:
- 障害扱いしない
- valid result として手入力導線を提示

UI 表示:
- 「見つかりませんでした。手入力で登録しますか？」

### E-03 API 429 / 5xx / キー無効

システム対応:
- 自動 retry 上限後、検索機能のみ劣化
- 既存蔵書・予約・執行・記録は通常継続
- 限定告知に留める

UI 表示:
- 「現在、書籍情報の検索機能のみ利用しづらい状態です。手入力で登録できます」

### E-04 オフラインで本追加したい

システム対応:
- 検索 UI は unavailable 表示
- 手入力登録を許可
- 後続 metadata 補完は任意・非 blocking

### E-05 書影画像がない

発生例:
- `thumbnailUrl` 欠損
- 画像取得失敗
- cache miss + オフライン

システム対応:
- app bundle の placeholder 画像を表示
- タイトル・著者を主表示に切替
- 書影の欠損を理由に登録・選択・開始を止めない
- disk cache に以前の書影があればそれを優先

UI 表示:
- 無言で placeholder に差し替える
- エラートーストは原則出さない

止めてはいけないもの:
- 本登録
- Focus Book 表示
- Widget 表示
- セッション開始

記録:
- `cover_placeholder_used`

### E-06 タイトルはあるが著者・ページ数・価格がない

システム対応:
- 欠損許容で Book 保存
- 著者不明は「著者不明」
- ページ数不明は「ページ数不明」
- 価格不明なら資産化は pageCount fallback、それも無理なら金額非表示

### E-07 同じ本が重複候補として出る

システム対応:
- `sourceVolumeId` 優先で集約
- 欠ける場合は `title + author` 正規化で二次判定
- UI では重複候補を減らす

### E-08 候補を選んだのに保存失敗

システム対応:
- 1 回だけ再試行
- 失敗なら「手入力で保存」または「あとでやり直す」を提示
- 画面クラッシュさせない

## 3.2 Focus Book / 計画生成 / Reconcile

### E-09 今日の plan が存在しない

システム対応:
- `ReconcilePlansUseCase` で当日 plan を補完生成
- それでも作れない場合はホームで「今日のセッションを作成する」を主 CTA とする

### E-10 plan はあるが `bookId` が壊れている / 対応 book がない

システム対応:
- Strategic Bypass または Library から代替本を選ぶ
- 本が 1 冊もなければ AddBook 導線へ切替
- 壊れた参照のままホームを描画しない

UI 表示:
- 「今日の本を再設定しましょう」

### E-11 連続未達日数が不整合

システム対応:
- 日次締めを再実行し再計算
- `continuous_missed_days` は表示値でなく履歴から再導出可能にする

### E-12 日付跨ぎ / 04:00 ロールオーバーの catch-up 失敗

システム対応:
- 次回 app launch / foreground / widget / notification response で再試行
- 冪等に再実行可能とする

### E-13 Focus Book 自動選定に必要な metadata が足りない

システム対応:
- 欠損フィールドだけで脱落させない
- 優先順位は known data だけで評価
- どうしても選べない場合は最近追加本または未読本から安全な既定を選ぶ

## 3.3 通知

### E-14 通知権限が拒否されている

システム対応:
- ホーム固定 CTA
- Widget 開始ボタン
- App Intents / Shortcuts
- アプリ内リマインドバナー
- 権限再要求は乱発しない

UI 表示:
- 「通知なしでも使えます。必要なら後でオンにできます」

### E-15 通知予約に失敗する

システム対応:
- pending request を再取得して再整合
- request ID ベースで不足分のみ再構築
- app 内主導線は維持

### E-16 通知が重複している

システム対応:
- 安定 ID で照合し重複削除
- 同日撤退時の通知 suppression を再適用

### E-17 通知アクション応答時に plan が stale

システム対応:
- 開始前に必ず reconcile
- stale のまま startSession しない

### E-18 通知から開始したが session 開始直前に失敗

システム対応:
- app を開いて同じ plan に対する再試行導線を出す
- 二重 session 作成を避ける

### E-19 再通知がしつこくなる

システム対応:
- 上限 1 回固定
- 連投禁止
- 再通知不能でも miss 判定へ自然遷移

---

## 3.4 Widget / Live Activity / App Intents

### E-20 Widget の状態が stale

システム対応:
- 表示前 reconcile
- shared snapshot 更新失敗時は最後の安全な snapshot かプレーン表示へ縮退
- stale でも誤開始しないよう start 時再度 reconcile

### E-21 Widget に書影が出ない

システム対応:
- placeholder 表示
- タイトル優先レイアウトに切替

### E-22 Widget から開始できない

システム対応:
- app 起動にフォールバック
- app 内で startSession 再試行導線を表示

### E-23 Live Activity が開始できない

システム対応:
- セッション本体は継続
- 進行 UI をアプリ画面に限定
- Live Activity 失敗をセッション失敗扱いしない

### E-24 Live Activity が更新停止 / 終了漏れ

システム対応:
- app foreground 復帰時に再同期
- 完了時は best-effort で終了
- stale 表示が残っても正本は SwiftData を優先

### E-25 App Intent から起動したが shared snapshot が壊れている

システム対応:
- snapshot を信頼せず、必ず native 正本 or reconcile 経由で開始判定

---

## 3.5 セッション開始・進行・完了

### E-26 セッション開始時に二重タップ / 多重起動

システム対応:
- `StartSessionUseCase` を冪等化
- 同一 plan の active session があれば再利用
- 二重作成しない

### E-27 セッション中にアプリが落ちる

システム対応:
- `startedAt` / `endTimeISO` を正本に先書き
- 次回起動時に `RestoreActiveSessionUseCase` で復旧

### E-28 セッション中にアプリを閉じる / OS が kill する

システム対応:
- 正本から active session 復元
- Live Activity の有無に依存しない

### E-29 セッション完了直前で保存失敗

システム対応:
- 完了処理を再試行可能にする
- 冪等に result を確定
- 二重完了・二重加算を避ける

### E-30 1 分救済と `prep_success` の意味が混線する

システム対応:
- 1 分で実際にセッション完了したなら実績として扱う
- ただし読了申告や資産化増分の扱いは別途ルール化
- 「継続維持」「実績」「読了申告」を別保存する

### E-31 中断したが attempted か missed か曖昧

システム対応:
- `ATTEMPT_MINIMUM_SECONDS` 以上なら attempted 候補
- それ未満で未完了なら missed または cancelled_system を判定ルールで確定

### E-32 セッション中に時計変更 / タイムゾーン変更

システム対応:
- remaining は monotonic 基準優先
- 帰属日は startedAt と rollover ルールで確定
- 端末時計変更だけで結果を二重反転させない

---

## 3.6 完了後フロー・資産化

### E-33 価格情報がない

システム対応:
- `pageCount * DEFAULT_ASSET_VALUE_PER_PAGE_YEN` を使う
- それも無理なら金額表示なしで前進メッセージへフォールバック

### E-34 ページ数も価格もない

システム対応:
- 資産額は非表示
- 「前に進みました」系メッセージで継続（標準フォールバック）
- 完了後フロー自体は止めない

### E-35 読了申告が保存できない

システム対応:
- セッション完了自体は確定済みにする
- `読了した` による状態更新だけ再試行可能にする
- 一般完了では自己申告保存自体を発生させない

### E-35A プログレスバーが未設定 / OFF / 入力欠損

システム対応:
- 初回セッション完了前は設定案内を出さない
- 初回完了後のみソフト案内し、拒否時は再提示を抑制する
- 総ページ数は metadata 自動取得を優先し、欠損時は任意入力へフォールバック
- 現在ページ未入力でも完了処理・達成表示を継続する
- いつでも OFF に戻せる状態を維持する

止めてはいけないもの:
- セッション開始 / 完了
- 完了後の達成体験
- 翌日の再開導線

### E-36 読了処理後の次本指名に失敗

システム対応:
- 読了状態は保存
- 次本指名はスキップ可能
- 次回ホームで再提案

---

## 3.7 外部読書アプリ

### E-37 電子書籍アプリ deep link が開けない

システム対応:
- セッション開始は維持
- 「読書アプリを開けませんでした。アプリを切り替えて続けてください」程度の軽い案内のみ
- deep link 失敗で session を取り消さない

### E-38 外部アプリ起動後の行動を観測できない

システム対応:
- 仕様上問題なし
- 読書成功条件を外部アプリ観測へ依存させない

---

## 3.8 永続化・データ保全

### E-39 SwiftData 書き込み失敗

システム対応:
- 可能なら 1 回だけ再試行
- 失敗した entity 単位でエラー隔離
- plan / session の保存失敗は Level A として扱う

### E-40 一部データ破損

システム対応:
- 破損行だけ隔離
- 参照不能 book を除外して再計算
- 必要なら当日 plan 再生成
- 全件初期化は最後の手段

### E-41 one-shot migration 失敗

システム対応:
- 二重書き込みに戻さない
- 移行未完として旧 read path 参照または migration 再試行
- ユーザーには「データを確認しています」程度に留める

### E-42 backup / restore 後の通知不整合

システム対応:
- 起動時 reconcile で通知再構築
- 通知センター残骸を信頼しない

---

## 3.9 CloudKit / 同期

### E-43 iCloud 無効

システム対応:
- ローカル継続を優先
- Premium sync だけ unavailable 扱い

### E-44 CloudKit 同期遅延

システム対応:
- ローカル正本で継続
- 同期は追いつけばよい
- 状態遷移を止めない

### E-45 マルチデバイス競合

システム対応:
- 日次計画と active session は競合時ルールを定義
- 同日同 plan の二重 active を避ける
- 競合解決不能時は last writer ではなく domain rule 優先

---

## 3.10 entitlement / 課金

### E-46 entitlement 最新化に失敗

システム対応:
- 直前の既知状態を暫定利用
- 無料機能を不必要にロックしない
- 課金確認失敗で読書機能本体を止めない

### E-47 購入完了通知が遅れる

システム対応:
- restore / refresh 導線を用意
- 反映遅延メッセージを出す

### E-48 課金系 UI が壊れている

システム対応:
- 執行機能には影響させない
- premium 訴求だけ抑制

---

## 3.11 EventLog / 可観測性

### E-49 EventLog 書き込み失敗

システム対応:
- 本体処理を止めない
- best-effort 再試行またはメモリ退避
- 計測不能と体験停止を混同しない

### E-50 同一イベント重複記録

システム対応:
- 冪等 key で重複抑制
- 集計時に二重加算しない

---

## 3.12 権限・端末制約

### E-51 通知権限はあるが Focus / おやすみで見えづらい

システム対応:
- 未応答時の再通知は上限内で実施
- app 内・Widget・Shortcut の代替起点を維持

### E-52 低メモリで書影 cache が消える

システム対応:
- memory cache miss は正常系扱い
- disk cache / placeholder へフォールバック

### E-53 OS バージョン差異で一部機能不可

システム対応:
- Live Activity / interactive widget / App Intents は availability check
- 非対応端末では app 内主導線へ縮退

## 3.13 セキュリティ・設定不備

### E-54 API キー設定漏れ

システム対応:
- 検索機能のみ unavailable
- 手入力導線を維持
- 既存蔵書の執行は通常通り

### E-55 App Group 設定不備で WidgetSnapshotStore に書けない

システム対応:
- Widget / Live Activity / App Intents だけ縮退
- アプリ本体は通常継続
- 開発・QA では release blocker 扱い

### E-56 通知カテゴリ / action ID の設定漏れ

システム対応:
- 通知タップ時は app 起動にフォールバック
- ただし CTA 動作不全は release blocker 扱い

## 4. ユーザー向け表示原則

### 4.1 基本

- 障害内容を技術用語で出しすぎない
- 原因追及より、次にできることを出す
- 罪悪感を増やす文言は禁止
- 書影欠損のような軽微異常は無言縮退を基本にする

### 4.2 推奨表現

- 「通信エラーにより検索できませんでした。手入力で登録できます」
- 「通知なしでも使えます。必要なら後でオンにできます」
- 「今日はこのまま始められます」
- 「書籍情報の一部が取得できませんでしたが、登録は続けられます」
- 「資産額は表示できませんが、今日の記録は残っています」

### 4.3 禁止表現

- 「失敗しました」だけで終わる
- 「原因不明です」だけで終わる
- 「やり直してください」だけで次導線がない
- 「通知を許可しないと使えません」
- 「書影がないため登録できません」

## 5. release blocker と non-blocker の線引き

## 5.1 release blocker

- セッション開始できない
- セッション完了記録が安定しない
- active session 復旧できない
- plan 再整合で日次状態が壊れる
- 通知開始時に別 plan / 別 book が起動する
- 旧系と新系の二重 write が残る
- App Group / shared snapshot 不整合で OS surface の開始結果が本体と一致しない

## 5.2 non-blocker だが要観測

- 書影欠損
- 価格欠損
- pageCount 欠損
- metadata 補完失敗
- EventLog 一部欠損
- CloudKit 一時遅延
- Live Activity 表示失敗（本体継続可能な場合）

---

## 6. QA 観点の最低テストセット

### 6.1 本追加

- オフライン時に手入力できる
- 書影なし候補を登録できる
- pageCount なしでも保存できる
- timeout 後に手入力へ行ける

### 6.2 ホーム / 計画

- book 参照欠損時に代替導線へ落ちる
- 当日 plan 欠損時に補完生成される
- 04:00 跨ぎでも state が再整合する

### 6.3 通知 / OS surface

- 通知拒否でもホーム・Widget・Shortcut で始められる
- stale widget でも start 時に reconcile される
- Live Activity 失敗でもセッション継続する

### 6.4 セッション

- 多重開始しても二重 session が作られない
- クラッシュ後に active session を復旧できる
- 完了保存失敗時でも再試行で二重加算しない

### 6.5 補助要素

- 価格なしで資産額 fallback する
- 価格もページ数もないとき金額非表示になる
- entitlement 未取得でも無料機能は使える

本書は以下に使う。

- 論理仕様書における異常時の補完規約
- UI/UX 指針のエラーメッセージと縮退運転ルールの根拠
- 実装時の fallback 優先順位
- QA の異常系テストケース設計
- release blocker 判定

更新単位は機能単位ではなく、**ユーザー価値を止める異常の種類単位**とする。
