## 1. プロダクトの定義

本アプリは「読書管理アプリ」ではない。
本アプリは、**その日の読書達成を成立させる執行支援アプリ。開始行動はそのための最小起点**である。

### 1.1 解決したい課題

ユーザーは本を所有していても、以下の理由で読まない。

* 読む意思決定の初速が重い
* 「今日は無理」の1回で習慣が切れる
* 未達の自己嫌悪によってアプリ自体を開かなくなる
* 本の管理はしても、読む行為の開始にはつながらない

### 1.2 MVP の価値仮説

以下が成立すれば MVP は成立とみなす。

* ユーザーが「今日の1冊」と「今日の開始時刻」を1タップ前後で確認できる
* 定刻時に読む行為を開始できる
* 未着手時にハードルを下げて救済できる
* 1日失敗しても翌日に再開しやすい
* 数日離脱しても復帰導線がある

## 2. 設計原則

### 2.1 観測可能な事実だけを扱う

本アプリが観測できるのは以下であり、「本当に読んだか」は直接観測できない。

* 通知を配信したか
* 通知に応答したか
* セッションを開始したか
* セッションが完了したか
* ユーザーが読了を申告したか
* ユーザーがページ進捗を更新したか（任意）

したがって、本仕様では「サボり」は扱わない。扱うのは次のみ。

* `未応答`
* `未着手`
* `延期`
* `中断`
* `未達`
* `再開`

### 2.2 状態とイベントを分離する

* 状態: 現在システムがどこにいるか
* イベント: 状態を変化させるトリガー

### 2.3 成功判定と読了申告を分離する

以下は別概念として保持する。

* **執行結果**: システムが判定する成功/未達
* **読了申告**: ユーザーが「読了した」を明示する入力
* **継続維持**: 今日とのつながりを切らさなかったか
* **ページ進捗**: progress tracking 利用時のみ扱う任意入力

`prep_success` は継続維持には使ってよいが、読書進捗には使わない。
**メンツは保つが、進捗には嘘をつかない** を原則とする。
「1分で再点火（Ignition）」は `prep_success` を得るための最小セッションとして扱い、
完了時はその日の継続維持を保証する。

### 2.4 Decision Fatigue を最小化する

MVP で重要なのは、ユーザーに「選ばせる」ことではなく、**提示された Focus Book と開始時刻を承諾させること**である。

* 執行の主導線では「本を選ぶ」を要求しない
* 迷う余地がある操作は library / 設定 / 完了後フローへ退避する
* 通知と執行前 UI の主責務は、選択肢提示ではなく開始承諾である

### 2.5 MVP では「本を大量に登録すること」より「今日読むこと」を優先する

MVP で重要なのは蔵書管理の網羅性ではなく、**最初の1冊を最短で今日の執行対象にすること**である。

### 2.6 MVP は Local First で完結させる

MVP の主要ロジックは **端末内で完結** させる。

* 自前バックエンド、API サーバー、認証基盤は持たない
* メインアプリの UI とユースケース実行は `React Native + TypeScript` で構成する
* `Book`、`DailyExecutionPlan`、`SessionLog`、設定情報は端末内ストアに保存する
* 通知は Push 通知ではなくローカル通知を使う
* 日次締めや通知再整合は `BackgroundTasks` とアプリ起動時の補正で実行する
* データ保全は `SwiftData` と iOS 標準バックアップ経路を前提に担保する
* Expo managed workflow は前提にしない。custom native module と app extension を許容する bare React Native 構成とする

### 2.7 Local First で負う責任

自前バックエンドを持たない以上、端末側で以下を **必須で担保** する。

* データ永続性: `SwiftData` と iOS 標準のバックアップ経路を前提に、端末故障や機種変更でも復元可能性を持たせる
* 通知信頼性: ローカル通知の request を厳密に再整合し、欠落・重複・古い時刻の残存を防ぐ
* 劣化運用: iCloud 同期遅延や BG task の遅延があっても、ローカル単体で状態遷移を継続できる

### 2.8 日次確定は Reconcile 方式で行う

* 04:00 の単発バッチ成功には依存しない
* stale plan の締め、欠損日補完、当日 plan 生成、通知再整合、active session 復旧は `ReconcilePlansUseCase` がまとめて担う
* 発火点は app launch / foreground / notification response / widget 描画 / App Intents / background task とする

### 2.9 外部 API 利用と自前バックエンドを混同しない

書籍検索で `Google Books API` のような外部 API を使うことは、**自前バックエンドを持つこととは別** である。

* 通信経路は `iOS app / React Native app -> HTTPS -> Google Books API` とする
* 書誌候補の取得はクライアントから直接行い、MVP では自前プロキシを挟まない
* ユーザーが選択した候補だけを正規化し、local の `SwiftData` へ保存する
* 検索通信は登録導線の補助であり、コア状態管理や日次判定をサーバー依存にしない

### 2.10 アーキテクチャ収束と移行規約

MVP を削らずに成立させるため、以下を固定する。

* 正本は `SwiftData`。状態主体は `DailyExecutionPlan`。行動記録は `SessionLog`。読書対象は `Book`。
* OS 接点（通知 / Widget / Live Activity / App Intents）は `WidgetSnapshotStore` 相当の shared read model を参照する。
* 起点の統一は `ReconcilePlansUseCase` と開始系 use case で行う。
* 旧 AsyncStorage 系への **新規 write は禁止**。旧導線は移行専用 read path としてのみ残し、cutover 後に削除する。
* 画面は business rule を持たず、use case 呼び出しに限定する。
* 必要時のみ one-shot migration（AsyncStorage -> SwiftData）を実施し、恒久的な二重書き込みは行わない。

* 翌日の再予約導線
* 連続未達時の再開導線

* 全ユーザー横断の集計分析

## 4. 用語定義

### 4.1 主要用語

* **Book**
  ユーザーが読む対象として登録した本

* **Focus Book**
  現在の執行対象として、ユーザーに継続提示する本

* **Daily Execution Plan**
  その日の読書執行計画。状態管理の主体

* **Session**
  1回の執行試行。通常 15 分、救済時 3 分または 5 分

* **Soft Success**
  救済モードまたは短時間でも完了した成功

* **Hard Success**
  通常セッションを完了した成功

* **Prep Success**
  本を机に置く、開く直前まで持っていく等、読書開始の摩擦を下げる準備が完了した状態

* **Attempted**
  セッションは開始したが完了しなかった

* **Missed**
  当日計画があったが未着手または未完了で終了した

* **Cancelled**
  本が手元にない等、実行不能理由が明示された終了

* **Rehab Mode**
  連続未達時にハードルを下げて再開しやすくする状態

* **Strategic Bypass**
  習慣の断絶を防ぐため、一定条件で一時的に軽量本へ Focus Book を切り替える挙動

* **Continuity Streak**
  `hard_success` / `soft_success` / `attempted` / `prep_success` により維持される継続指標

* **Completion Feedback**
  完了後に表示する短い前進フィードバック。実行時間、必要時の進捗率、次の行動を返す

## 5. システム構成上の前提

### 5.1 データ保存

MVP の永続データは **端末内保存を第一選択** とする。

* React Native 側は `PersistenceBridge` 経由で永続層へアクセスする
* native iOS 側の実装第一候補は `SwiftData`
* `Core Data` は native iOS 層で移行や制約対応が必要な場合の代替
* 自前サーバー DB は持たない
* 単一ユーザー前提であり、ログインを要求しない
* `active` 遷移時の `startedAt` は UI 遷移前に永続化し、クラッシュ復旧の truth source とする

### 5.2 通知

通知は `UserNotifications` による **ローカル通知** で端末内スケジュールする。([Apple Developer][2])

* サーバーからの Push 通知は MVP では使わない
* 通知予約はアプリ起動時、foreground 復帰時、widget 描画時、設定変更時、対象本変更時、セッション完了時、日次締め時に再整合する
* 通知 request は `planId + notificationType + scheduledAt` を元にした安定 ID で管理する
* スケジューリング後は pending request を再取得し、期待件数と識別子一致を検証する
* アプリ状態を唯一の truth source とし、通知センター上の残存 request を信頼しすぎない

### 5.3 日次締めとバックグラウンド処理

業務日境界は **04:00** とし、`day_rollover` は端末内で処理する。

* `BackgroundTasks` を使い、アプリが閉じていても締め処理を試行する
* iOS の都合で 04:00 ちょうどの実行は保証されないため、04:00 以降の最初の起動 / foreground / BG task 実行時に catch-up できればよい
* 日次締めの判定は「現在時刻が日境界を越えたか」で行い、サーバー時刻には依存しない
* Widget の描画時も `ReconcilePlansUseCase` を通し、前日未確定計画の締めと今日表示の整合を取る

### 5.4 iCloud / Cloud Sync

Local First を崩さずにデータ保全と同期を両立する。

* 無料層の基準は local 保存と iOS 標準バックアップである
* Premium 層では `CloudKit` 連携による multi-device sync を提供してよい
* これは自前バックエンドの代替ではなく、ユーザー個人領域でのデータ保全 / 同期手段である
* iCloud 未設定や同期遅延があっても、アプリはローカル単体で継続利用できる

### 5.5 Widget / Live Activity

Live Activities はロック画面、Dynamic Island、StandBy などで進行中データを表示するための仕組みであり、WidgetKit / ActivityKit 系で扱う。([Apple Developer][3])

* React Native が直接描画するのではなく、native iOS extension / target で実装する
* extension 側には `WidgetSnapshotStore` などの共有 read model を渡す
* React Native 本体と extension 間の共有には App Group 共有領域または同等の shared container を使う

### 5.6 Focus 連携

Focus 連携は App Intents の Focus filter 文脈で実装可能だが、本アプリでは **必須依存にはしない**。任意の補助連携とする。([Apple Developer][4])

### 5.7 外部読書アプリ

外部読書アプリを「本の特定位置」で開く挙動は、MVP の保証対象外とする。
可能な場合にアプリ起動導線を提供してもよいが、**読書成功条件に含めない**。

ただし外部読書アプリへ遷移する場合は、以下を必須とする。

* 遷移前に 5 秒間の ritual countdown、または同等の短い精神統一 UI を挟む
* その場の遷移だけを成功扱いしない
* 深い位置連携や外部アプリ内行動の観測には依存しない

### 5.8 外部書誌 API と API キー管理

書誌検索は `Google Books API` をクライアントから直接呼び出してよい。

* `BookSearchService` は `https://www.googleapis.com/books/v1/volumes?q=...` 相当へ HTTPS リクエストを送る
* 外部通信が必要なのは本検索と任意の metadata 補完だけであり、執行開始・タイマー・完了記録には使わない
* MVP では自前バックエンドやプロキシを設けず、検索結果 JSON を端末側で解釈する
* API キーは React Native の TypeScript ソースへ平文で直書きしない
* API キーは native iOS 側の build setting / config file で管理し、必要時のみ app 層へ渡す
* Google Cloud Console 側で **iOS Bundle ID 制限を必須** とする
* キー隠蔽は完全防御ではないため、軽い難読化はしてよいが、前提とすべき防御は利用元制限である
* オフライン、API 制限、レスポンス異常時でも登録導線を止めず、手動入力 fallback を許可する

### 5.9 外部 API 通信異常時の縮退運転

外部 API の異常は検索機能だけを劣化させ、読書執行そのものは止めない。

* 検索要求発火後 `BOOK_SEARCH_REQUEST_TIMEOUT_SECONDS` 秒以内に正常応答しない場合は timeout とみなす
* timeout、オフライン、HTTP 429、HTTP 5xx、API キー無効時は `BOOK_SEARCH_AUTO_RETRY_COUNT` 回だけ自動 retry し、それでも失敗なら手動入力導線へ切り替える
* 検索中 indicator は `BOOK_SEARCH_SPINNER_MAX_SECONDS` を超えて表示しない
* 検索結果 0 件は障害ではなく valid result として扱い、「見つかりませんでした。手入力で登録しますか？」を提示する
* `thumbnailUrl` 欠損時は app 内 placeholder 画像を表示し、`pageCount` 欠損時は `undefined` のまま保存して UI では「ページ数不明」と表示する
* 登録後の metadata 補完通信が失敗しても blocking dialog を出さず、既存 local 値または fallback 値で継続表示する
* API キー失効や外部 API 停止時は「現在、書籍情報の検索機能のみメンテナンス中です」と限定告知し、既存蔵書の予約・執行・記録は通常通り継続する
* `DailyExecutionPlan`、`SessionLog`、通知、Live Activity の更新は外部通信完了を待たずに local だけで確定する

## 6. コアエンティティ

MVP の状態管理は以下 4 エンティティを中心に行う。

1. `UserSettings`
2. `Book`
3. `DailyExecutionPlan`
4. `SessionLog`

### 6.1 状態管理の主体

**状態の主体はアプリ全体ではなく `DailyExecutionPlan` とする。**

理由:

* 日ごとの成功 / 未達 / 救済を管理しやすい
* 通知・スヌーズ・執行・再開のイベントを 1 集約に閉じ込められる
* 分析がしやすい

以下は MVP の固定値とする。

```text
DEFAULT_SESSION_MINUTES = 15
RESCUE_SESSION_MINUTES_PRIMARY = 5
RESCUE_SESSION_MINUTES_REHAB = 3
PRE_NOTICE_MINUTES = 10

IGNITION_SESSION_MINUTES = 1
DUE_RESPONSE_WINDOW_MINUTES = 15
DAY_ROLLOVER_HOUR = 4
REHAB_THRESHOLD_DAYS = 3
RESET_PLAN_THRESHOLD_DAYS = 7
ATTEMPT_MINIMUM_SECONDS = 60
MAX_DAILY_MANUAL_FOCUS_CHANGE = 1

BOOK_SEARCH_RESULT_LIMIT = 5

BOOK_SEARCH_MEMORY_CACHE_MINUTES = 5

BOOK_SEARCH_SPINNER_MAX_SECONDS = 10
BOOK_SEARCH_AUTO_RETRY_COUNT = 1
EBOOK_LAUNCH_RITUAL_SECONDS = 5
LIGHTWEIGHT_BOOK_PAGE_THRESHOLD = 180
RECENTLY_ADDED_BOOK_DAYS = 14
PREMIUM_TRIAL_DAYS = 7
```

### 7.1 補足

* `DAY_ROLLOVER_HOUR = 4` は業務日境界。深夜の実行を前日計画に帰属させやすくするため
* 日次締めは 04:00 ちょうどの実行を前提にせず、04:00 以降の catch-up 実行を許容する
* `BOOK_SEARCH_DEBOUNCE_MILLISECONDS = 300` は検索負荷と入力追従性のバランスを取るため
* `BOOK_SEARCH_RESULT_LIMIT = 5` は候補選択に必要十分な件数の目安
* `BOOK_SEARCH_MEMORY_CACHE_MINUTES = 5` は検索語の微修正時に無駄な API 呼び出しを抑えるため
* `BOOK_SEARCH_REQUEST_TIMEOUT_SECONDS = 5` は検索通信が遅延しても登録導線を詰まらせないための一次 timeout
* `BOOK_SEARCH_SPINNER_MAX_SECONDS = 10` は検索中 indicator の上限
* `BOOK_SEARCH_AUTO_RETRY_COUNT = 1` は transient failure に対する隠れた救済回数
* `IGNITION_SESSION_MINUTES = 1` は「本を開く前の摩擦除去」に特化した最小実行単位
* `ATTEMPT_MINIMUM_SECONDS = 60` は開始直後離脱をノイズ扱いしないための下限
* `MAX_DAILY_MANUAL_FOCUS_CHANGE = 1` は Focus Book の主体性を確保しつつ再選択疲れを防ぐため
* `LIGHTWEIGHT_BOOK_PAGE_THRESHOLD = 180` は Strategic Bypass 時に「軽い本」を選ぶ目安
* `RECENTLY_ADDED_BOOK_DAYS = 14` は最近追加本を優先候補へ含める期間
* `PREMIUM_TRIAL_DAYS = 7` は `subscription` 提供時の初回体験期間既定値（`one_time` では未使用）

## 8. 状態モデル

## 8.1 アプリ準備状態

| 状態            | 意味       |
| ------------- | -------- |
| `needs_setup` | 初回設定未完了  |
| `ready`       | 初回設定完了済み |

## 8.2 日次執行状態 (`DailyExecutionPlan.state`)

| 状態            | 意味            |
| ------------- | ------------- |
| `unscheduled` | 当日計画なし        |
| `scheduled`   | 当日計画あり、定刻前    |
| `due`         | 定刻到来、応答待ち     |
| `deferred`    | 再通知待ちまたは明示延期後 |
| `active`      | セッション進行中      |
| `finalized`   | 当日の計画が確定終了    |

補足:

* `finalized` は通知運用上の終了を意味する
* ただし `DAY_ROLLOVER_HOUR` までは、同一 `planDate` に対する追加セッション記録を許可する
* 追加セッション記録後は `DailyExecutionPlan.result` を再集計し、格上げのみ許可する

## 8.3 セッションモード (`SessionLog.mode`)

| 値               | 意味            |
| --------------- | ------------- |
| `normal_15m`    | 通常 15 分       |
| `rescue_5m`     | 通常時の軽量開始 / 追加セッション / OS surface の軽量開始 |
| `rehab_3m`      | 3日以上未達時の再開補助（Rehab） |
| `ignition_1m`   | `prep_success` 取得のための最小セッション（再点火） |

## 8.4 執行結果 (`DailyExecutionPlan.result`)

| 値                                     | 意味             |
| ------------------------------------- | -------------- |
| `hard_success`                        | 通常セッション完了      |
| `soft_success`                        | 救済セッション完了      |
| `prep_success`                        | 準備だけ完了し、開始摩擦を下げた |
| `attempted`                           | 開始したが完了せず終了    |
| `missed`                              | 未応答・未開始・未完了で終了 |
| `cancelled_book_unavailable`          | 本が手元にないため終了    |
| `cancelled_user_paused_for_today`     | 明示延期が上限に達し、その日の追跡を終了 |
| `cancelled_user_deferred_to_tomorrow` | 明日へ回すことを明示選択   |
| `cancelled_system`                    | システム都合の異常終了    |

## 8.5 読了申告 (`DailyExecutionPlan.progress_self_report`)

| 値                | 意味     |
| ---------------- | ------ |
| `none`           | 未入力    |
| `finished_book`  | 読了した   |

## 8.6 プログレスバー運用規約（任意補助）

* 初回セッション完了前は、プログレスバー設定を尋ねない
* 初回セッション完了後にのみ、ソフト案内で有効化を提案してよい
* プログレスバーは任意機能であり、未設定でも主導線（開始・継続・完了）を阻害しない
* 総ページ数は書誌 metadata 自動取得を優先し、欠損時のみ任意入力を受ける
* 現在ページ入力は任意とし、毎回更新を必須にしない
* ユーザーはいつでも OFF に戻せる
* OFF 時はページ入力 UI を隠し、達成体験の文言・導線を維持する

## 9. 日付帰属ルール

### 9.1 セッションの帰属日

セッションは **開始時刻ベース** で当日計画に帰属する。

例:

* 2026-03-09 23:58 に開始 → 2026-03-09 の計画に帰属
* 2026-03-10 00:10 に開始、かつロールオーバー前 → 2026-03-09 の計画に帰属可能
* 2026-03-10 04:01 以降 → 2026-03-10 の日付として扱う

### 9.2 日次締め処理

日次締めは、`ReconcilePlansUseCase(referenceDate)` によって行う。

この処理は以下の契機で実行する。

* アプリ起動時
* アプリ foreground 復帰時
* 通知アクション受信時
* Widget / Siri / Shortcuts / App Intents 実行時
* Background Task 実行時（best-effort）

処理内容:

1. 最終確定済み日以降、`referenceDate - 1日` までの欠損日を探索する
2. その日に計画が存在し、かつ未確定なら `finalized(result=missed)` として確定する
3. その日に計画が存在しないが、継続運用上は計画があるべき場合は、補完用 `DailyExecutionPlan` を生成し `finalized(result=missed)` とする
4. 当日分の `DailyExecutionPlan` が存在しない場合は、`FocusBookSelector` により Focus Book を決定して自動生成する
5. `continuous_missed_days` を再計算する
6. 通知予約を再整合する

設計原則:

* 正しさを 04:00 のバッチ実行成功に依存させない
* ローカル通知は先に予約し、レコード確定は後から整合してよい
* widget には、reconcile 済みの「今日の1冊」と状態だけを表示する
* Focus Book は前日継承を既定とし、必要時のみ Strategic Bypass または manual override を適用する

### 9.3 未予約日

その日に `DailyExecutionPlan` が作られていない場合、その日は `missed` に数えない。

## 10. 状態遷移概要

## 10.1 正常系

```text
needs_setup
  └─ registration_complete → ready + DailyExecutionPlan(scheduled)

scheduled
  └─ timer_fired → due

due
  └─ session_start_manual / session_start_from_notification / session_start_from_widget → active

active
  └─ session_completed(normal_15m) → finalized(hard_success)
  └─ session_completed(rescue_5m / rehab_3m) → finalized(soft_success)
```

## 10.2 救済系

```text
due
  └─ session_start_from_notification / session_start_manual / session_start_from_widget → active(normal_15m)
  └─ rescue_selected_5m → active(rescue_5m)
  └─ rehab_selected_3m → active(rehab_3m)
  └─ snooze_selected → deferred
  └─ hidden_secondary_action → focus_book_changed / not_now_selected / prep_only_selected
active(ignition_1m)
  └─ prep_completed → finalized(prep_success)
```

## 10.3 未応答系

```text
due
  └─ response_timeout (retry_count = 0) → deferred(retry pending)
deferred
  └─ retry_timer_fired → due
due
  └─ response_timeout (retry_count = 1) → finalized(missed)
```

## 10.4 中断系

```text
active
  └─ session_cancelled(elapsed >= 60s) → finalized(attempted)
  └─ session_cancelled(elapsed < 60s) → finalized(missed)
```

## 10.5 finalized 後の追加セッション

```text
finalized(soft_success / attempted / prep_success)
  └─ extra_session_started → finalized + SessionLog(open)

finalized + extra SessionLog(completed)
  └─ daily_result_recomputed → finalized(result upgraded only)
```

## 10.6 復旧系

```text
active
  └─ reconcile_plans + open session found → active(restored)
finalized
  └─ reconcile_plans + open extra session found → finalized(restored extra session)
  └─ reconcile_plans + elapsed >= planned duration → finalized(derived from session completion)
```

## 11. イベント仕様

各イベントは以下を持つ。

* 発火条件
* 許可状態
* payload
* 副作用
* 遷移先
* 冪等性

## 11.1 オンボーディング系イベント

### `registration_complete`

* 発火条件:

  * 初回設定で以下が揃った

    * 最初の1冊
    * 通常開始時刻
    * 通知権限結果
* 許可状態:

  * `app_state = needs_setup`
* payload:

  * `book_id`
  * `preferred_start_time`
  * `notifications_permission`
* 副作用:

  * `UserSettings` 作成
  * `Book` 作成
  * 当日または次回分の `DailyExecutionPlan` 作成
* 遷移:

  * `app_state: needs_setup -> ready`
  * `plan.state: unscheduled -> scheduled`
* 冪等性:

  * 同日同一ユーザーに対して複数回実行されても、最新設定で upsert

---

## 11.2 スケジュール系イベント

### `reconcile_plans`

* 発火条件:

  * app launch / foreground / background task / widget render / notification response / app intent
* 許可状態:

  * 全状態
* payload:

  * `executed_at`
  * `reference_date`
  * `trigger_source = app_launch | foreground | background_task | widget_render | notification_response | app_intent`
* 副作用:

  * stale plan の締め
  * 欠損日 plan の補完
  * 当日 plan の存在確認と必要時の生成
  * active session の復旧判定
  * 通知予約の再整合
* 遷移:

  * stale な `scheduled`, `due`, `deferred` は必要に応じて `finalized(missed)` へ
  * `active` は復旧または完了判定へ進む
* 冪等性:

  * 同一時点で複数回走っても整合性を壊さない

### `schedule_created`

* 発火条件:

  * 当日の計画を新規作成
* 許可状態:

  * `plan.state = unscheduled`
* payload:

  * `plan_date`
  * `scheduled_at`
  * `book_id`
  * `planned_duration_minutes`
* 副作用:

  * ローカル通知をスケジュール
  * 10 分前予告通知をスケジュール
  * 予約済み通知との重複を解消
* 遷移:

  * `unscheduled -> scheduled`
* 冪等性:

  * 同日計画が既にある場合は upsert

### `schedule_updated`

* 発火条件:

  * 時刻変更、対象本変更、予定時間変更
* 許可状態:

  * `scheduled`, `due`, `deferred`
* payload:

  * `scheduled_at`
  * `book_id`
  * `planned_duration_minutes`
* 副作用:

  * 既存 pending local notification を取り消し、再スケジュール
* 遷移:

  * 状態維持
* 冪等性:

  * 同一 payload の重複適用は no-op

### `day_rollover`

* 発火条件:

  * `reconcile_plans` 実行中に 04:00 以降の日境界越えを検知
* 許可状態:

  * 全状態
* payload:

  * `executed_at`
  * `trigger_source`
* 副作用:

  * 当日未確定計画の締め
  * 翌日の計画生成判定
  * `continuous_missed_days` 更新
  * 通知予約の再整合
* 遷移:

  * `scheduled`, `due`, `deferred` は `finalized(missed)` へ
* 冪等性:

  * 同一業務日に対して 1 回のみ有効

---

## 11.3 通知・定刻系イベント

### `pre_notice_fired`

* 発火条件:

  * 定刻 10 分前
* 許可状態:

  * `scheduled`
* payload:

  * なし
* 副作用:

  * 予告通知配信
* 遷移:

  * 状態維持
* 冪等性:

  * 1 回限り

### `timer_fired`

* 発火条件:

  * 定刻到来
* 許可状態:

  * `scheduled`
* payload:

  * `scheduled_at`
* 副作用:

  * 本通知配信
  * Live Activity 開始準備
* 遷移:

  * `scheduled -> due`
* 冪等性:

  * 同一計画に対して 1 回限り

### `retry_timer_fired`

* 発火条件:

  * 未応答再通知の再定刻
* 許可状態:

  * `deferred` かつ `defer_reason = unresponsive_retry`
* payload:

  * なし
* 副作用:

  * 再通知配信
* 遷移:

  * `deferred -> due`
* 冪等性:

  * `retry_count < MAX_UNRESPONSIVE_RETRY_COUNT + 1` の範囲のみ

### `response_timeout`

* 発火条件:

  * `due` 開始から `DUE_RESPONSE_WINDOW_MINUTES` 経過
* 許可状態:

  * `due`
* payload:

  * なし
* 分岐:

  * `retry_count = 0` の場合:

    * `due -> deferred`
    * `defer_reason = unresponsive_retry`
    * `retry_count += 1`
    * `UNRESPONSIVE_RETRY_DELAY_MINUTES` 後に再通知予約
  * `retry_count >= MAX_UNRESPONSIVE_RETRY_COUNT` の場合:

    * `due -> finalized(missed)`
* 冪等性:

  * タイムアウト済みフラグがあれば no-op

---

## 11.4 開始系イベント

### `session_start_from_notification`

* 発火条件:

  * 通知アクション「今すぐ読む」
* 許可状態:

  * `due`, `deferred`
* payload:

  * `entry_point = notification`
* 副作用:

  * SessionLog 作成
  * `SessionLog.startedAt` と `DailyExecutionPlan.startedAt` を永続化
  * Live Activity 開始
* 遷移:

  * `due/deferred -> active`
* 冪等性:

  * 既に `active` が存在する場合は no-op

### `session_start_from_widget`

* 発火条件:

  * Widget の開始ボタン押下
* 許可状態:

  * `scheduled`, `due`, `deferred`
* payload:

  * `entry_point = widget`
* 副作用:

  * 早期開始なら通知を整理
  * SessionLog 作成
  * `SessionLog.startedAt` と `DailyExecutionPlan.startedAt` を永続化
  * Live Activity 開始
* 遷移:

  * `scheduled/due/deferred -> active`
* 冪等性:

  * 既に `active` が存在する場合は no-op

### `session_start_manual`

* 発火条件:

  * アプリ内開始ボタン押下
* 許可状態:

  * `scheduled`, `due`, `deferred`
* payload:

  * `entry_point = app`
* 副作用:

  * SessionLog 作成
  * `SessionLog.startedAt` と `DailyExecutionPlan.startedAt` を永続化
  * Live Activity 開始
* 遷移:

  * `scheduled/due/deferred -> active`
* 冪等性:

  * 既に `active` が存在する場合は no-op

## 11.5 救済・延期イベント

### `rescue_selected_5m`

* 発火条件:

  * 「5分だけ」選択
* 許可状態:

  * `due`
* payload:

  * `mode = rescue_5m`
* 副作用:

  * 5 分 SessionLog 作成
  * Live Activity 開始
* 遷移:

  * `due -> active`
* 冪等性:

  * active なら no-op

### `rehab_selected_3m`

* 発火条件:

  * Rehab 提案画面から「3分だけ」選択
* 許可状態:

  * `due`
* payload:

  * `mode = rehab_3m`
* 副作用:

  * 3 分 SessionLog 作成
  * Live Activity 開始
* 遷移:

  * `due -> active`
* 冪等性:

  * active なら no-op

### `snooze_selected`

* 発火条件:

  * 「30分延期」選択
* 許可状態:

  * `due`
* payload:

  * なし
* 前提:

  * `snooze_count < MAX_SNOOZE_COUNT`
* 副作用:

  * 30 分後に通知を予約
  * `snooze_count += 1`
* 遷移:

  * `due -> deferred`
* 冪等性:

  * 既に同一スヌーズ予約があれば no-op

### `not_now_selected`

* 発火条件:

  * 二次操作として「今日はやめる」を選択
* 許可状態:

  * `due`, `deferred`
* payload:

  * なし
* 分岐:

  * `explicit_defer_count + 1 < MAX_EXPLICIT_DEFER_COUNT` の場合:

    * `explicit_defer_count += 1`
    * 最後の確認通知を 1 回だけ予約
    * `due/deferred -> deferred`
  * `explicit_defer_count + 1 >= MAX_EXPLICIT_DEFER_COUNT` の場合:

    * `explicit_defer_count += 1`
    * その日の pending 通知を全取消
    * 「承知しました。本日はゆっくりお休みください」を表示
    * `due/deferred -> finalized(cancelled_user_paused_for_today)`
* 冪等性:

  * finalized 後は no-op

### `book_unavailable_selected`

* 発火条件:

  * 二次操作として「本が手元にない」を選択
* 許可状態:

  * `due`
* payload:

  * なし
* UI 遷移:

  * 次の二択のみ表示する
  * `open_ebook_app_selected`
  * `prep_only_selected`
* 制約:

  * この分岐内では `defer_to_tomorrow_selected` を表示しない
* 冪等性:

  * finalized 後は no-op

### `open_ebook_app_selected`

* 発火条件:

  * 「電子書籍を開く」選択
* 許可状態:

  * `due`
* payload:

  * optional: `target_app`
* 副作用:

  * `EBOOK_LAUNCH_RITUAL_SECONDS` 秒の ritual countdown を表示
  * 設定済みの電子書籍アプリを best-effort で開く
  * SessionLog 作成
  * `SessionLog.startedAt` と `DailyExecutionPlan.startedAt` を永続化
* 遷移:

  * `due -> active`
* 備考:

  * 特定位置の再開は保証しない
* 冪等性:

  * active なら no-op

### `prep_only_selected`

* 発火条件:

  * 「1分で再点火（Ignition）」または「準備だけする（1分）」選択
* 許可状態:

  * `due`
* payload:

  * なし
* 副作用:

  * `IGNITION_SESSION_MINUTES` 分タイマー開始
  * SessionLog 作成
  * `SessionLog.startedAt` と `DailyExecutionPlan.startedAt` を永続化
* 遷移:

  * `due -> active`
* 冪等性:

  * active なら no-op

### `prep_completed`

* 発火条件:

  * 1 分再点火タイマー完了
* 許可状態:

  * `active` かつ `mode = ignition_1m`
* 副作用:

  * `result = prep_success`
  * `consistencyCredit = true`
  * `continuity_streak_days` 維持対象として確定
* 遷移:

  * `active -> finalized`
* 冪等性:

  * 同一 session の完了処理は 1 回のみ

### `defer_to_tomorrow_selected`

* 発火条件:

  * 「今日はやめて明日にする」選択
* 許可状態:

  * `due`, `deferred`
* payload:

  * なし
* 副作用:

  * 翌日の同時刻計画を既定値で提示
* 遷移:

  * `finalized(cancelled_user_deferred_to_tomorrow)`
* 冪等性:

  * finalized 後は no-op

## 11.6 セッション進行イベント

### `session_completed`

* 発火条件:

  * セッションタイマー完了
* 許可状態:

  * `active`
  * `finalized` かつ open extra session が存在する
* payload:

  * `session_id`
  * `elapsed_seconds`
* 判定:

  * `mode = normal_15m` → `hard_success`
  * `mode = rescue_5m / rehab_3m` → `soft_success`
* 副作用:

  * Live Activity 終了または通常セッションに引継ぎ
  * 完了入力 UI を表示
  * 追加セッションが存在する場合は `daily_result_recomputed` を発火する
* 遷移:

  * 通常/救済完了 → `active -> finalized`
  * 追加セッション完了 → `finalized -> finalized`
* 冪等性:

  * 同一 session の完了処理は 1 回のみ

### `session_cancelled`

* 発火条件:

  * 中断ボタン押下、またはセッション破棄
* 許可状態:

  * `active`
  * `finalized` かつ open extra session が存在する
* payload:

  * `elapsed_seconds`
* 判定:

  * `plan.state = active` の場合:

    * `elapsed_seconds >= ATTEMPT_MINIMUM_SECONDS` → `attempted`
    * `elapsed_seconds < ATTEMPT_MINIMUM_SECONDS` → `missed`
  * `plan.state = finalized` の場合:

    * 追加セッションの中断として記録する
    * `DailyExecutionPlan.result` は維持し、格下げしない
* 副作用:

  * Live Activity 終了
* 遷移:

  * `active/finalized -> finalized`
* 冪等性:

  * finalized 後は no-op

### `progress_self_reported`

* 発火条件:

  * 完了後に `読了した` を選択
* 許可状態:

  * `finalized` かつ `result in (soft_success, hard_success, attempted)`
* payload:

  * `finished_book`
  * optional: `current_page`
  * optional: `total_pages`
* 副作用:

  * 読了申告保存
  * 読了時は Book を `completed` に更新
  * 読了時はお祝い演出と次の Focus Book 指名フローを開いてよい
* 遷移:

  * 状態維持
* 冪等性:

  * 上書き可

### `progress_tracking_enabled`

* 発火条件:

  * 初回完了後の opt-in 案内で有効化を選択
* 許可状態:

  * `finalized`
* payload:

  * `enabled = true`
  * optional: `source = post_completion | book_detail`
* 副作用:

  * `UserSettings.progressTrackingEnabled = true`
* 遷移:

  * 状態維持

### `progress_tracking_skipped`

* 発火条件:

  * 初回完了後の opt-in 案内で「あとで / 使わない」を選択
* 許可状態:

  * `finalized`
* payload:

  * `reason = later | disabled`
* 副作用:

  * 進捗バー設定画面を閉じる
* 遷移:

  * 状態維持

### `book_progress_updated`

* 発火条件:

  * 本詳細または完了後導線で現在ページを更新
* 許可状態:

  * 全状態
* payload:

  * `book_id`
  * optional: `current_page`
  * optional: `page_count`
* 副作用:

  * `Book.currentPage` / `Book.pageCount` / `Book.lastProgressUpdatedAt` 更新
* 遷移:

  * 状態維持

### `extra_session_started`

* 発火条件:

  * 完了画面またはホームから「もう 5 分 / 15 分」選択
* 許可状態:

  * `finalized` かつ `result in (soft_success, attempted, prep_success)`
* payload:

  * `mode = normal_15m | rescue_5m`
* 前提:

  * 現在時刻 < `DAY_ROLLOVER_HOUR` の日次境界
* 副作用:

  * 既存結果は破壊的に上書きせず、新しい `SessionLog` を追加作成
  * UI 上は「格上げ」に見えてよいが、内部では session の積み増しとして保持する
  * `SessionLog.startedAt` と `DailyExecutionPlan.startedAt` を永続化
* 遷移:

  * `finalized` のまま
* 冪等性:

  * 同一操作の重複受付は no-op

### `daily_result_recomputed`

* 発火条件:

  * 追加セッション完了
* 許可状態:

  * `finalized`
* 副作用:

  * 当日全 `SessionLog` を再評価
  * `DailyExecutionPlan.result` を格上げのみ更新
* 制約:

  * `hard_success -> soft_success` のような格下げは禁止
* 冪等性:

  * 同一 session 群に対する再計算は同じ結果に収束する

---

## 11.7 システム例外イベント

### `active_session_restored`

* 発火条件:

  * `reconcile_plans` 実行時に `plan.state in (active, finalized)` かつ `endedAt` 未設定の SessionLog を検出
* 許可状態:

  * `active`
  * `finalized`
* payload:

  * `executed_at`
  * `elapsed_seconds`
* 分岐:

  * `elapsed_seconds < planned_duration_minutes * 60` の場合:

    * 残り時間を再計算してタイマー UI を復旧
  * `elapsed_seconds >= planned_duration_minutes * 60` の場合:

    * `session_completed` 相当の完了処理を適用
* 副作用:

  * Live Activity が消えていれば再開可能な範囲で再構成
* 遷移:

  * `active` 維持または `finalized`
* 冪等性:

  * 同一 session に対する復旧処理は 1 回の結果に収束する

### `notification_permission_denied`

* 発火条件:

  * 通知権限拒否
* 許可状態:

  * 任意
* payload:

  * なし
* 副作用:

  * アプリ内バナーと設定導線表示
  * Widget/Siri 開始を強く案内
* 遷移:

  * 状態維持
* 冪等性:

  * 毎回通知不要、1 日 1 回以下でよい

### `focus_book_changed`

* 発火条件:

  * Focus Book を変更
* 許可状態:

  * `unscheduled`, `scheduled`, `due`, `finalized`
* payload:

  * `book_id`
  * `source = home_ghost_link | library`
* 副作用:

  * 当日または次回の計画へ反映
  * `DailyExecutionPlan.focusStrategy = manual_override`
  * `manualFocusChangeCount += 1`
* 制約:

  * `active` 中は変更禁止
  * `manualFocusChangeCount < MAX_DAILY_MANUAL_FOCUS_CHANGE` の範囲でのみ許可
  * 上限到達時は `focus_book_changed` を受け付けず、Ghost Link は非表示または disabled 表示
  * ホームでは書影下の Ghost Link からのみ起動する
  * 通知や DueActionSheet の主導線には直接置かない
* 遷移:

  * 状態維持
* 冪等性:

  * 同一 book なら no-op

### `next_focus_book_nominated`

* 発火条件:

  * 読了後フローで次の1冊を確定
* 許可状態:

  * `finalized`
* payload:

  * `book_id`
* 副作用:

  * 次の Focus Book を更新
  * 必要に応じて翌日計画へ反映
* 遷移:

  * 状態維持
* 冪等性:

  * 同一 book なら no-op

## 12. 日次判定ルール

## 12.1 成功判定

### `hard_success`

以下の両方を満たす。

* 通常 15 分セッションを完了
* `session_completed` が記録された

### `soft_success`

以下のいずれかを満たす。

* 救済 5 分セッションを完了
* Rehab 3 分セッションを完了

### `prep_success`

以下を満たす。

* 「1分で再点火（Ignition）」または準備行動を完了した
* 本を読む準備行動を明示記録した
* 読書開始摩擦を下げたことは認める
* その日の継続維持は成功として扱う
* 読書進捗グラフの増分には使わない
* `consistencyCredit = true` の対象にしてよい

### `attempted`

以下を満たす。

* セッション開始済み
* `elapsed_seconds >= ATTEMPT_MINIMUM_SECONDS`
* 完了前に中断

### `missed`

以下のいずれかを満たす。

* `due` で未応答のまま再通知上限を超過
* セッション未開始のまま日次締め
* セッション開始後 60 秒未満で中断

### `cancelled_*`

以下のいずれかを満たす。

* 本が手元にないと明示
* 明日へ回すと明示
* その日の追跡停止を明示

## 12.2 `continuous_missed_days` 更新ルール

対象は **DailyExecutionPlan が存在した日だけ** とする。

| 当日結果           | 更新           |
| -------------- | ------------ |
| `missed`       | `+1`         |
| `hard_success` | `0` にリセット    |
| `soft_success` | `0` にリセット    |
| `prep_success` | `0` にリセット    |
| `attempted`    | `0` にリセット    |
| `cancelled_*`  | 原則 `0` にリセット |
| 計画なし           | 変動なし         |

理由:

* 「やれなかったが戻る意思はある」ケースを missed 連鎖に含めないため

## 12.3 継続評価と進捗評価の分離

### 継続評価 (`continuity_streak_days`)

以下の日は継続を維持したものとして扱ってよい。

* `hard_success`
* `soft_success`
* `attempted`
* `prep_success`

### 読書進捗評価

以下のみが読書進捗の増分対象である。

* `hard_success`
* `soft_success`
* `progress_self_reported`

以下は **増分対象にしない**。

* `prep_success`
* `attempted`

原則:

* `prep_success` は自己嫌悪を避けるための継続維持ロジックであり、読んだことにはしない
* ストリークは守ってよいが、進捗グラフには嘘をつかない

## 12.4 結果優先順位

日次結果の再集計では、以下の優先順位で **格上げのみ** 許可する。

```text
hard_success > soft_success > attempted > prep_success > cancelled_* > missed
```

## 13. Focus Book / Rehab Mode 仕様

## 13.1 Focus Book 基本方針

* `DailyExecutionPlan` 生成時、前日の Focus Book を既定で自動継承する
* ユーザーが読了するか library で明示変更するまで、同じ本を継続提示する
* 執行フローの入り口では「本を選ばせる」より「この本で始める」を優先する

## 13.2 Focus Book の手動変更（最小自由度）

方針:

* ユーザーは Focus Book を手動変更する権利を持つ
* ただし再選択疲れを避けるため、`MAX_DAILY_MANUAL_FOCUS_CHANGE = 1` を既定とする
* ホームでは書影下の Ghost Link「本を切り替える」からのみ変更させる
* 主 CTA は常に開始系を維持し、手動変更は主導線を邪魔しない配置に固定する

反映:

* 同日内では `focusStrategy = manual_override` を最優先する
* 通知レスポンス導線や DueActionSheet の主 CTA には「本を変える」を置かない

## 13.3 Strategic Bypass（戦略的迂回）

発動条件:

* `continuous_missed_days >= 3`

挙動:

* 当日の Focus Book を一時的に軽量本へ切り替えて提示してよい
* 軽量本の判定は、`pageCount` が少ない本、または最近追加された本を優先する
* `DailyExecutionPlan.focusStrategy = strategic_bypass` として記録する

目的:

* 重い本への心理抵抗で習慣が死ぬのを防ぐ
* 読書純度よりストリーク回復を優先する

## 13.4 Rehab Mode 発火条件

### レベル 1

* `continuous_missed_days >= 3`

### レベル 2

* `continuous_missed_days >= 7`

## 13.5 Rehab Mode 振る舞い

### レベル 1 (`>= 3`)

ホームの主 CTA を **1分で再点火** に切り替える。
副 CTA は **5分だけ** を維持し、3分は Rehab バナーで再開提案として出す。
必要に応じて Strategic Bypass による軽量本切替を伴ってよい。

### レベル 2 (`>= 7`)

アプリ起動時、通常ホームより先に **再開専用導線** を表示する。

内容:

* library を開く
* 時間を変える
* 1分だけで再開する
* 今日はやめる

## 13.6 禁止事項

* 責める文言
* 罪悪感を煽る表現
* 「連続失敗」自体を強調する UI
* 執行の主導線で本選択を要求する UI
* 未達後も重い本を惰性で出し続けること

## 14. UI/UX 制約

## 14.1 オンボーディング

要件:

* 3 ステップ以内で完了する
* 最初の1冊を登録する
* 通常開始時刻を設定する
* 通知権限を要求する

MVP の登録方法:

* インクリメンタル検索
* 候補は外部書誌 API から取得する
* 候補が得られない場合のみ手動入力にフォールバックする
* 手動入力では `title` 必須、`author` / `pageCount` / `cover image` は任意とする

インクリメンタル検索要件:

* 入力中に `BOOK_SEARCH_DEBOUNCE_MILLISECONDS` 程度で候補更新する
* 候補 5 件程度で十分とする
* 候補行には最低でもタイトル、著者、書影サムネイルを出す
* `Google Books API` の `volumes?q=...` 相当の外部書誌検索を使ってよい
* 検索はクライアントから直接 HTTPS で実行してよく、自前バックエンドを前提にしない
* 検索結果キャッシュは最低 `BOOK_SEARCH_CACHE_TTL_HOURS` 保持する
* 短期 memory cache を `BOOK_SEARCH_MEMORY_CACHE_MINUTES` 程度保持し、検索語の微修正で毎回 API を叩かない
* 選択済み書影は次回起動時や Widget 表示でも使えるよう disk cache を持つ
* 検索結果の `author`、`subtitle`、`thumbnailUrl`、`pageCount`、`estimatedPriceAmount` は欠損を許容する
* 重複候補は `sourceVolumeId` を主キーとして best-effort で集約し、欠ける場合は正規化した `title + author` で二次判定する
* 候補選択後は結果を local の `Book` へ正規化して保存し、検索レスポンス全体をそのまま保存しない
* 検索要求後 `BOOK_SEARCH_REQUEST_TIMEOUT_SECONDS` 秒で 1 回だけ自動 retry し、それでも失敗なら「通信エラーにより検索できませんでした。手入力で登録しますか？」を表示する
* loading indicator は表示してよいが、`BOOK_SEARCH_SPINNER_MAX_SECONDS` を超えて待たせない
* `0件` の場合は「見つかりませんでした。手入力で登録しますか？」を表示する
* `thumbnailUrl` がない候補でも `NO IMAGE` placeholder で選択可能にする
* `pageCount` がない候補でも登録可能にし、表示は「ページ数不明」とする
* オフライン時も最低限手動入力で登録できる

非対応:

* OCR
* ISBN
* 一括登録

## 14.2 ホーム

表示要素:

* 今日の1冊
* 次の開始時刻
* 主 CTA（状態に応じた第一導線）
* 5 分だけボタン（副 CTA）
* 書影下の Ghost Link「本を切り替える」（1 日 1 回まで）
* 連続未達時は Rehab バナー
* `heavy_day_signal = true` 時は軽量トーンの補助文

原則:

* CTA は常に 1 つを最上位に置く
* 情報量を増やしすぎない
* 通常時の主 CTA は 15 分開始、副 CTA は 5 分だけ
* `continuous_missed_days >= 3 && < 7` は主 CTA を「1分で再点火」、副 CTA を「5分だけ」とし、3分再開は Rehab バナーで提示する
* `continuous_missed_days >= 7` は通常ホームより先に再開専用導線を表示し、主 CTA を「1分だけで再開する」に固定する
* 「1分で再点火（Ignition）」は `prep_success` の取得導線であり、継続維持に使う
* 「本を切り替える」は Ghost Link として控えめに配置し、主導線に昇格させない

## 14.3 執行画面

表示要素:

* 本タイトル
* 残り時間
* 中断ボタン
* 最小限の補助文言

禁止:

* 本棚一覧への遷移
* 統計の深掘り導線
* 読書中に判断を増やす UI

## 14.4 完了後フロー

表示要素:

* セッション結果に応じた前進メッセージ
* 実行時間
* お祝い演出（読了時）
* 「もう 5 分やる」
* 「もう 15 分やる」
* 「読了した」
* 初回のみ progress tracking opt-in 導線

注意:

* 一般前進はユーザー入力ではなく完了メッセージで返す
* `progress_self_reported` は `finished_book` のみで使う
* これは**執行結果の上書きではない**

追加導線:

* `soft_success` / `attempted` / `prep_success` 後は「もう 5 分やる」「もう 15 分やる」の追加セッション CTA を出してよい
* UI 上は結果の格上げに見せてよいが、内部では `SessionLog` を追加する
* `finished_book` 後は「次の Focus Book を指名する」導線を即時提示する
* 次本指名は library / queue から 1 冊だけ承諾させる形を既定とする

## 14.5 完了後フィードバック

目的:

* 読書完了後に、前進感を短く返す
* 義務感ではなく、達成感と次の行動を与える

表示優先順位:

1. 実行時間（必須）
2. 読了率または進捗率（progress tracking 有効時かつ取得可能な場合）
3. 前進メッセージ

補助規約:

* 金額換算は行わない
* progress tracking が OFF または未設定でも完了体験を成立させる
* `prep_success` は progress 増分に使わない

実装上の注意:

* `pageCount` / `currentPage` は optional 扱いとする
* ページ情報がなければ前進メッセージへフォールバックする
* 長い分析は完了直後画面に持ち込まない

## 14.6 コミュニケーショントーン

原則:

* 責めない
* しつこくない
* 引き際を持つ

具体ルール:

* 二次操作の「今日はやめる」が 2 回選ばれた日は、その日の追跡を打ち切る
* 打ち切り時は「承知しました。本日はゆっくりお休みください」と表示する
* その後は同日内に追撃通知を送らない
* 詳細な表現規範は `28. コミュニケーション・ガイドライン` に従う

## 14.7 外部 reader への遷移

* 電子書籍アプリへ handoff する前に `EBOOK_LAUNCH_RITUAL_SECONDS` 秒のカウントダウンを入れる
* 文言は「スマホを伏せてください。読書を開始します」等の短い儀式でよい
* handoff だけで成功扱いしない

## 14.8 Library / Queue

目的:

* Focus Book の変更、2冊目以降の追加、待機列の確認を執行導線から分離する

原則:

* 通知から開始までの主導線に library を差し込まない
* 「本を変える」は基本的に library の責務とし、ホームでは Ghost Link からの最小導線のみ許可する
* 読了後の次本指名は library / queue を軽量に開いて 1 冊承諾させる

## 15. 通知仕様

## 15.1 通知種別

1. 予告通知
2. 本通知
3. 未応答再通知
4. 翌日再予約確認通知
5. Rehab 再開提案通知

## 15.2 通知ルール

共通ルール:

* 全通知はローカル通知で実装し、サーバー Push は使わない
* pending request はアプリ起動時、foreground 復帰時、widget 描画時、設定変更時、対象本変更時、セッション完了時、`day_rollover` 後に再整合する
* 予約対象は当日計画と次回計画に必要な通知に限定し、重複予約を避ける
* 各通知 request は安定 ID を持ち、同一計画の再スケジュール時は取消後に再作成する
* 再整合処理では「期待される request 集合」と「実際の pending request 集合」を比較し、不一致があれば再構築する
* 通知センター内に存在しても、計画状態と不整合な request は無条件で破棄する
* 明示延期が `MAX_EXPLICIT_DEFER_COUNT` に達した日は、その日の残り通知を全取消する

### 予告通知

* 定刻 10 分前
* 文言例: 「あと10分で開始です。本の準備を」

### 本通知

* 定刻
* アクション:

  * 開始
  * 5分だけ
  * 30分延期
* `開始` は状態依存で第一導線へ map する（通常: 15分、3日以上未達: 1分、7日以上未達: 1分）
* 通知タップ後の `DueActionSheet` は「開始」を主 CTA とし、「5分だけ」「30分延期」を同階層の副選択に置く
* 「本を変える」「今日はやめる」は二次階層へ退避し、開始判断の主導線を汚さない
* Focus Book の変更はホームの Ghost Link または library からのみ実行可能とする

### 未応答再通知

* 15 分後に 1 回のみ
* しつこい連投は禁止

### 二次操作の扱い

* 「本を変える」「今日はやめる」は本通知アクションに置かない
* これらは DueActionSheet の二次階層または library から実行させる

### 翌日再予約確認

* 執行完了直後または当日終了時に提示
* デフォルトは前回時刻

## 15.3 通知拒否時の代替導線

* ホームの固定 CTA
* Widget 開始ボタン
* Siri / Shortcuts 開始
* アプリ内リマインドバナー

## 16. OS Surface 仕様（責務固定）

OS 接点は周辺機能ではなく MVP の受入対象である。  
優先順は `通知 -> Widget -> Live Activity -> App Intents -> 電子書籍導線` とする。

## 16.1 通知（Notification）

表示責務:

* 定刻通知、再通知（上限あり）、予告通知を表示する
* 本通知アクションは `開始 / 5分だけ / 30分延期` の 3 つに固定する

開始責務:

* 通知応答で開始可能とする
* 応答処理は必ず `ReconcilePlansUseCase` 実行後に進める
* `開始` は状態依存で第一導線へ map する（通常: 15分、`continuous_missed_days >= 3`: 1分、`>= 7`: 1分）

許可 CTA:

* 開始
* 5分だけ
* 30分延期

禁止 CTA:

* 本変更
* Library 遷移
* 管理操作（設定変更など）

## 16.2 Widget（Interactive Widget）

表示責務:

* 今日の1冊
* 今日の状態（開始可否）
* 開始ボタンと 5分だけボタン

開始責務:

* Widget から開始できる
* 表示前・開始前の双方で `ReconcilePlansUseCase` を通す
* `開始` は状態依存で第一導線へ map する（通常: 15分、`continuous_missed_days >= 3`: 1分、`>= 7`: 1分）

許可 CTA:

* 開始
* 5分だけ

禁止 CTA:

* 本変更
* 長い設定導線

インタラクティブ Widget と Live Activity のアクション連携は WidgetKit / App Intents の文脈で成立する。([Apple Developer][5])

## 16.3 Live Activity

位置付け:

* **補助機能**
* コア状態管理を依存させない

表示責務:

* 執行中の残り時間
* 対象 book title
* 実行中であることの明示
* `normal_15m / rescue_5m / rehab_3m / ignition_1m` の各セッションを区別して表示できること

開始責務:

* `session_start_*` の副作用として開始
* セッション更新で残時間を更新
* 完了 / 中断 / 異常終了で終了

許可 CTA:

* 進行確認（必要最小限）

禁止 CTA:

* Focus Book の変更
* 結果確定を伴う操作

Live Activities はロック画面や Dynamic Island 等で現在進行中データを表示する用途に使う。([Apple Developer][3])

## 16.4 App Intents（Siri / Shortcuts / Controls）

表示責務:

* 状態表示は shared read model から取得する

開始責務:

* App Intent 経由開始時も `ReconcilePlansUseCase` を先行させる
* `AppIntentStartSessionUseCase` に委譲する
* `読書開始` は状態依存で第一導線へ map する（通常: 15分、`continuous_missed_days >= 3`: 1分、`>= 7`: 1分）

許可 CTA:

* 読書開始
* 5分だけ読む
* 今日の本確認

禁止 CTA:

* 状態機械を直接書き換える操作

App Intents は Siri や Shortcuts への露出基盤として利用できる。([Apple Developer][1])

## 16.5 電子書籍導線（Deep Link）

扱い:

* 「開始の代替」ではなく「開始の短縮」

順序:

1. `session_start_*` を先に確定
2. reader app への遷移は best-effort

禁止:

* reader 起動成功のみを成功扱いにすること

## 21. 受け入れ基準

## 21.1 実装完了判定

以下を満たせば MVP の論理仕様実装は完了とみなす。

* 初回設定完了で当日または次回の計画が作成される
* React Native 本体から native iOS の永続層・通知層へ bridge 経由でアクセスできる
* 本登録でインクリメンタル検索候補と書影サムネイルが表示される
* 書籍検索はクライアントから外部書誌 API を直接呼び出し、自前バックエンドなしで成立する
* 候補が得られない場合は手動入力にフォールバックできる
* 検索結果の欠損フィールドがあっても登録導線が壊れない
* 重複候補が返ってきても候補表示が過剰重複しない
* 書籍検索 timeout 後に 1 回だけ自動 retry し、それでも失敗なら手入力導線へ切り替わる
* 検索 loading indicator が `BOOK_SEARCH_SPINNER_MAX_SECONDS` を超えて残り続けない
* 検索結果 0 件でも空白で終わらず、手入力 CTA が表示される
* `thumbnailUrl` や `pageCount` が欠損していても登録を完遂できる
* 選択済み書影が再起動後も再利用される
* 主要データが自前サーバーなしで端末内保存される
* 前日の Focus Book が当日の計画へ既定で継承される
* `continuous_missed_days >= 3` で Strategic Bypass による軽量本推薦が起動できる
* 外部書誌 API キーが TypeScript ソースに平文で埋め込まれておらず、iOS Bundle ID 制限が前提になっている
* 定刻で通知が出る
* 通知から開始できる
* 本通知アクションが `開始 / 5分だけ / 30分延期` の 3 つに絞られている
* `DueActionSheet` が `開始` を主 CTA とし、`本を変える` が主導線に混ざらない
* 通常時ホームで `15分主 CTA / 5分副 CTA` が表示される
* `continuous_missed_days >= 3` ではホーム主 CTA が「1分で再点火（Ignition）」へ切り替わる
* `continuous_missed_days >= 3 && < 7` で `1分主 CTA / 5分副 CTA / 3分Rehab提案` が共存する
* `continuous_missed_days >= 7` では再開専用導線が先に表示され、主 CTA が 1 分になる
* `prep_success` が記録でき、継続維持と読書進捗が分離される
* 「1分で再点火（Ignition）」完了時、`prep_success` として継続が維持される
* 5分だけ導線が動く
* 未応答時に 15 分後再通知が 1 回だけ動く
* スヌーズが 1 回だけ動く
* セッション完了で `hard_success` / `soft_success` が正しく記録される
* 完了画面で `少し進んだ / かなり進んだ` が表示されない
* 完了画面で `もう 5 分やる / もう 15 分やる / 読了した` が表示される
* 一般完了で `progress_self_reported` を必須にしない
* `読了した` のみ次本指名分岐を起こす
* ページ進捗が未設定でも完了体験が成立する
* 完了後フィードバックはページ情報なしでも前進メッセージへフォールバックできる
* 中断で `attempted` / `missed` が正しく分岐する
* 翌日再予約導線が出る
* `continuous_missed_days` が正しく増減する
* 3 日連続未達で 3 分再開提案が出る
* 7 日連続未達で `RehabScreen` が最上位表示される
* `ReconcilePlansUseCase(referenceDate)` により欠損日補完・missed 確定・当日計画生成が集約される
* widget 描画時にも stale state が reconcile される
* アプリ再起動後も通知予約が再整合される
* 通知再整合時に期待 request 集合と pending request 集合の不一致が解消される
* Widget の user-facing CTA が `開始 / 5分だけ` に固定される
* App Intents の user-facing アクション名が `読書開始 / 5分だけ読む / 今日の本確認` に固定される
* Widget / Live Activity / App Intents が shared snapshot を通じて React Native 本体の状態と整合する
* 「今日はやめる」を 2 回選ぶと同日の追跡が停止する
* 完了後は金額換算を表示しない
* metadata 補完通信の失敗時も完了後フィードバックを blocking しない
* `finished_book` 後に Celebration と次の Focus Book 指名フローが起動する
* `LibraryScreen` から 2 冊目以降の追加と Focus Book 変更ができる
* 本詳細から表紙差し替え、タイトル/著者/ページ数修正、進捗設定、Focus Book 化ができる
* `HomeScreen` の Ghost Link から Focus Book を 1 日 1 回まで手動変更できる
* `finalized` 後の追加セッションで `SessionLog` が追加され、破壊的上書きにならない
* 追加セッション完了後に日次結果が格上げのみ再計算される
* 初回セッション完了後にのみプログレスバー有効化のソフト案内を出せる
* プログレスバーを未設定のままでも、開始・完了・達成表示が一切阻害されない
* 総ページ数は自動取得優先で、欠損時のみ任意入力になる
* 現在ページ入力は任意で、毎回更新を要求しない
* 本詳細から progress tracking の後追い設定・更新ができる
* プログレスバーをいつでも OFF に戻せる
* 5分セッションの開始 / 実行中 / 完了が一連で成立する
* active セッション中にアプリ再起動してもタイマーが復旧する
* 通知拒否時でもホーム / Widget / Siri から開始できる
* iCloud 同期が一時失敗しても主要フローが停止しない
* 外部書誌 API が停止または API キーが無効でも、既存本の予約・執行・記録が停止しない
* `StoreKit 2` 経由で Premium entitlement を判定できる（MVP は `one_time` 優先）
* Premium 有効時のみ `CloudKit` multi-device sync と分析レポート / 高度コーチ機能を解放できる

## 21.2 Premium / Entitlement

Premium は `StoreKit 2` を通じた entitlement として提供する。

* free: 基本の執行、通知、Focus Book 継続、救済モード
* premium: 高度な Rehab / Coach 提案、`CloudKit` multi-device sync
* MVP の販売モデルは `one_time` を優先する
* `subscription` は将来拡張として同一 entitlement モデルに追加可能とする
* `subscription` を提供する場合、初回体験期間は `PREMIUM_TRIAL_DAYS` を既定値としてよい
* 課金価値は「タイマー」ではなく、読書習慣の継続資産と再開支援に置く

* ホーム初回表示 300ms 以内を目標
* 開始操作から執行画面表示まで 500ms 以内を目標
* インクリメンタル検索の候補更新は入力停止後 500ms 以内を目標
* 検索中 indicator は `BOOK_SEARCH_SPINNER_MAX_SECONDS` 以内に成功・manual fallback・empty result のいずれかへ収束させる

### 22.2 信頼性

* 状態遷移は冪等
* 同一イベント重複でも整合性を壊さない
* 日次締めは再実行可能
* 04:00 以降の catch-up 実行でも同じ結果になる
* 通知予約は再整合可能で、重複予約を避ける
* 通知予約の欠落検知と再構築が可能である
* active session のクラッシュ復旧が可能である
* 外部書誌 API の欠損データ、重複候補、レート制限、オフラインを原因に登録不能へ直行しない
* 執行開始、タイマー進行、完了記録はネットワーク未接続でも 100% local で完結する
* 外部書誌 API の障害は検索系機能に限定して劣化し、既存本の執行フローへ波及しない

### 22.3 可観測性

* 全重要イベントを `EventLog` に記録（必須）
* 少なくとも以下を分析可能にする

  * 通知到達数
  * 開始率
  * 救済選択率
  * 完了率
  * 翌日継続率
  * 3 日後復帰率

MVP で記録必須のイベント名:

* `onboarding_started`
* `onboarding_completed`
* `book_added`
* `reservation_set`
* `notification_delivered`
* `notification_opened`
* `session_started`
* `session_completed`
* `result_finalized`
* `entitlement_viewed`
* `entitlement_purchased`

加えて以下を区別して計測可能にする。

* `prep_success` 率
* `consistency_rate`
* `reading_success_rate`
* progress tracking 利用率
* 完了後フィードバック到達率
* 書籍検索 timeout 率
* 手入力 fallback 遷移率
* Focus Book 承諾率
* trial -> premium 転換率
* premium 継続率

### 22.4 プライバシー

* 読書対象、実行履歴、未達情報は自前サーバーへ送信しない
* MVP の分析ログは端末内記録を前提とする
* 外部書誌 API へ送るのは検索語と書誌取得に必要な最小限データに限定する

### 22.5 データ保全

* 端末故障や機種変更に備え、`SwiftData` と iOS 標準バックアップ経路を利用する
* Premium では `CloudKit` を使った multi-device sync を提供してよい
* iCloud が無効でもローカル利用を阻害しない
* 同期遅延や一時失敗で状態遷移を止めない

### 22.6 メディアキャッシュ

* 選択済み書影は widget とアプリ双方で再利用できるよう disk cache を持つ
* 書影取得失敗時もタイトル選択フローを止めない

### 22.7 Local First の実装責任

* サーバーを持たない代わりに、データ消失リスク低減を端末実装の責任として負う
* 通知欠落リスク低減を端末実装の責任として負う
* React Native 本体と native extension / bridge の整合が取れない実装は MVP 完了と見なさない
* Focus Book 継承と通知再整合の両方が満たされない実装は MVP 完了と見なさない

### 22.8 外部 API セキュリティ

* 外部書誌 API キーは iOS Bundle ID 制限を前提に発行する
* API キーを TypeScript リポジトリへ平文コミットしない
* 軽い難読化は補助策として許容するが、主防御として過信しない

* 読書そのものの真偽は判定しない

* 「1ページだけ」は使わない
* 読書アプリとの深い同期はしない
* 通知・Widget・Siri のどれでも開始できれば十分とする
* ページ数入力はオプションであり、必須にしない
* プログレスバーは任意補助機能であり、未利用でもアプリ価値は変わらない

## 24. 実装時の禁止事項

* `rescue_selected` の時点で成功記録しない
* `idle` のような曖昧状態を復活させない
* 読了申告で執行結果を上書きしない

* `04:00 に必ず動くバッチ` を正解ロジックにしない
* 通知未応答時に複数回しつこく再通知しない
* 連続未達で責める文言を表示しない
* Live Activity をコア状態管理の唯一ソースにしない
* 当日計画がない日まで `continuous_missed_days` に含めない
* `finalized` の日次結果を破壊的更新で書き換えない
* 電子書籍アプリの handoff だけで成功扱いしない
* MVP で自前バックエンド、認証基盤、必須ログインを導入しない
* 個人の読書データを第三者サーバーへ送信する前提で設計しない
* `LibraryScreen` を通知から開始までの主導線に差し込まない

## 27. 最終要約

本アプリの MVP は、「読むかどうか」ではなく**今日の執行を開始させること**に最適化する。
実装方針は **Local First** とし、自前バックエンドを持たずに `React Native + TypeScript` をメインアプリ層、native iOS の `SwiftData` / ローカル通知 / `BackgroundTasks` / `StoreKit 2` / app extensions をシステム連携層として組み合わせて成立させる。
状態管理の主体は `DailyExecutionPlan` とし、イベント駆動で以下を成立させる。

* 定刻で始められる
* 前日の Focus Book を迷わず引き継げる
* 始められなければ軽くできる
* 失敗しても翌日に戻れる
* 連続未達でも Strategic Bypass と Rehab で再開できる
* 読了したらその場で次の1冊へ運用をつなげられる

通知再整合と Focus Book 継承は、Local First を成立させるための必須責務である。
さらに、候補選択中心の登録、`ReconcilePlansUseCase` による欠損日整合、`prep_success` による継続維持、Strategic Bypass、追加セッションの格上げ再集計を持つことで、心理的継続と実装堅牢性の両方を担保する。
Premium では、高度な再開提案と multi-device sync を追加価値として提供する。
それでも MVP の本質は **端末内で完結する執行支援** にある。

この原則を崩さない限り、MVP はぶれない。

## 28. コミュニケーション・ガイドライン

### 28.1 基本人格

* パフォーマンス・コーチとして沈着冷静（執事的な丁寧さは維持）
* 読書の実行に対しては一貫して実務的
* 挫折に対しては極めて寛容
* 鼓舞よりも、次の最小行動を提示する
* 理由を詰問しない

### 28.2 推奨表現

* 「1分で再点火しましょう。ここから戻せます」
* 「準備だけでも前進です」
* 「今日は軽く進めましょう」
* 「本を机に置けたら十分です」
* 「明日の開始時刻は、このままでよろしいですか」

### 28.3 禁止表現

* 「なぜ読まなかったのですか？」
* 「なぜやらないのですか？」
* 「頑張りましょう」
* 「今日も失敗ですね」
* 「連続未達です」
* 罪悪感を誘導する文言全般

### 28.4 コピー原則

* 責めない
* 理由を問い詰めない
* 選択肢は多くしすぎない
* 常に「今できる最小行動」を前面に出す
